## Key points to cover
- Explain precisely where converters look for the language: a class like `language-js` or `lang-python` on the inner `<code>` tag (this is the convention highlight.js, Prism, and most static-site generators use). If that class is missing or non-standard, the converter has nothing to map to a fenced code block's language hint, and it falls back to a plain ``` block.
- Cover the harder case: HTML generated by a syntax highlighter that applies colors via individual `<span>` tags around tokens, with no class identifying the language at all on the parent. There the language is genuinely unrecoverable from the markup — state this honestly rather than implying every converter "should" get it right.
- Give a concrete before/after: show the input `<pre><code class="language-js">`, the correct Markdown output ` ```js `, and the broken output ` ``` ` with no hint, so a reader can pattern-match their own case.
- Tie back to devmark's actual behavior: turndown + GFM plugin reads the `language-*` class when present; when the source has none, recommend the reader add it manually post-conversion rather than treating it as a bug.
